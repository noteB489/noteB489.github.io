요즘 "클로드(Claude)로 작업할 때는 markdown(md)보다 html이 낫다"는 말을 종종 본다.
핵심은 파일 확장자 자체보다, **AI가 문서 구조를 얼마나 안정적으로 이해하고 다룰 수 있느냐**에 있다.

## 1) "MD 대신 HTML"이 의미하는 것

이 표현은 보통 아래 의미를 포함한다.

1. 문서를 더 엄격한 구조로 다루고 싶다.
2. 섹션, 표, 주석, 데이터 속성 같은 메타 정보를 더 명시적으로 넣고 싶다.
3. 렌더링 결과(브라우저 출력)를 기준으로 작업 정확도를 높이고 싶다.

Markdown은 간결하고 빠르지만, 문법 해석이 도구마다 조금씩 다를 수 있다.
반면 HTML은 태그 구조가 명시적이어서 "어디가 제목/본문/컴포넌트인지"를 더 분명히 표현하기 쉽다.

## 2) 앞으로 어떻게 흘러갈까?

개인적으로는 "HTML로 완전 대체"보다는 **혼합 전략**이 늘어날 가능성이 크다고 본다.

- 글 중심 워크플로우: Markdown 유지
- UI/레이아웃/상호작용이 중요한 페이지: HTML(또는 JSX/컴포넌트) 강화
- AI 자동화 파이프라인: MD 원본 + HTML 빌드 산출물 병행

즉, 작성은 MD로 하고 배포나 검증 단계에서 HTML 구조를 적극 활용하는 방식이 현실적이다.

## 3) MD보다 HTML이 정말 효율적인가?

정답은 "작업 목적에 따라 다르다"이다.

### Markdown이 효율적인 경우

- 빠른 초안 작성
- 블로그/문서 위주의 콘텐츠 운영
- Git diff를 단순하게 유지하고 싶을 때

### HTML이 효율적인 경우

- 정교한 레이아웃 제어가 필요할 때
- 섹션별 메타데이터, 앵커, 속성 제어가 중요할 때
- AI에게 구조적 편집 지시를 반복적으로 줄 때

## 4) 실무 추천

1. **콘텐츠 원본은 MD로 유지**한다.
2. 꼭 필요한 페이지에서만 HTML 커스텀을 쓴다.
3. 팀 규칙을 만든다. (예: "포스트 본문은 MD, 랜딩 페이지는 HTML")
4. AI에게는 "수정 범위"를 명확히 준다. (예: `<section id='faq'>` 내부만 수정)

결론적으로, "MD vs HTML"은 우열보다는 역할 분담 문제다.
콘텐츠 중심 블로그라면 MD가 여전히 강력하고, 구조 제어가 중요한 페이지에서는 HTML이 확실한 장점이 있다.
